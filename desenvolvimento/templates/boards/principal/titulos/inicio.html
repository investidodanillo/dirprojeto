Tutorial Conceitual – Gestão de Desenvolvimento com o App "Desenvolvimento"

O aplicativo Desenvolvimento tem como objetivo sistematizar e organizar o processo de evolução dos sistemas criados a partir do MOLDE, permitindo o registro, acompanhamento e controle de tudo que envolve o ciclo de desenvolvimento.

1. Estrutura da Metodologia

O app é baseado em quatro pilares principais:

Sistema

Representa cada sistema/aplicativo que será gerenciado. Exemplo: MOLDE, Contabilidade, RH, Financeiro. É a unidade organizacional central, onde todos os itens de evolução, releases e checklists se conectam.

Item de Evolução

Representa uma demanda de desenvolvimento. Pode ser:

  • Nova Funcionalidade (Feature)
  • Correção de Bug
  • Melhoria
  • Documentação
  • Tarefa Técnica

Está sempre vinculado a um Sistema e pode, opcionalmente, estar associado a uma Categoria. Possui ciclo de vida com status (Pendente → Em Andamento → Bloqueado → Concluído).

Release

Representa uma versão de entrega de um sistema. Agrupa múltiplos Itens de Evolução concluídos em um marco único de entrega e documenta o que foi desenvolvido, corrigido ou melhorado em determinada versão.

Checklist

Conjunto de verificações para garantir a qualidade de uma entrega:

  • ChecklistSistema: lista de verificação completa (por exemplo, "Checklist de Publicação").
  • ChecklistItem: etapa individual (por exemplo, "Rodar migrações", "Executar testes unitários").

2. Fluxo de Trabalho

O fluxo recomendado segue etapas simples e organizadas:

  1. Registrar Sistemas
    Crie registros para cada sistema que será acompanhado (ex.: MOLDE, Contabilidade, Financeiro).
  2. Criar Itens de Evolução
    Toda nova demanda deve ser registrada como um Item de Evolução. Informe tipo, categoria (opcional), prioridade, e mantenha o status inicial como Pendente.
  3. Acompanhar Status
    Atualize o status conforme o item avança: Pendente → Em Andamento → Bloqueado → Concluído.
  4. Agrupar em Releases
    Quando várias demandas estiverem prontas, organize-as em uma Release (ex.: "v1.1 – relatório de centro de custos e correções").
  5. Validar com Checklists
    Antes de publicar a Release, execute os Checklists associados e marque cada ChecklistItem concluído.

3. Classificações Auxiliares

Para facilitar a organização, o app traz classificações auxiliares:

  • Categoria: agrupa os itens por área (ex.: Backend, Frontend, Banco de Dados).
  • TipoItem: define o tipo da demanda (FEATURE, BUG, MELHORIA, DOCUMENTACAO, TAREFA).
  • Prioridade: indica impacto/urgência (Baixa, Média, Alta, Crítica).
  • Status: mostra em que fase está o item (Pendente, Em Andamento, Bloqueado, Concluído).

Essas classificações permitem filtrar, priorizar e acompanhar o desenvolvimento de forma estruturada.

4. Cenários de Uso (Exemplos Práticos)

Cenário 1 — Nova Funcionalidade

Demanda: Relatório de Lançamentos Contábeis por Centro de Custos

  • Criar um Item de Evolução do tipo Feature.
  • Categoria: Relatórios / Frontend (se aplicável).
  • Sistema: Contabilidade.
  • Prioridade: Alta.
  • Versão Prevista: v1.1.
  • Acompanhar até Concluído e incluir na Release v1.1.
  • Executar Checklist de Publicação antes do deploy.

Cenário 2 — Correção de Erro

Demanda: Correção no cálculo de saldo acumulado

  • Criar um Item de Evolução do tipo Bug.
  • Categoria: Cálculos e Regras de Negócio.
  • Prioridade: Crítica.
  • Versão Prevista: v1.1 (ou Release emergencial v1.0.1 se necessário).
  • Executar Checklist de Testes de Segurança/QA.

Cenário 3 — Melhoria Contínua

Demanda: Otimização de consulta SQL lenta

  • Criar um Item de Evolução do tipo Melhoria.
  • Categoria: Backend / Performance.
  • Prioridade: Alta.
  • Planejar para a próxima Release e validar com checklist de performance.

5. Benefícios da Metodologia

  • Padronização: todos os desenvolvimentos seguem o mesmo fluxo.
  • Organização: demandas agrupadas por sistema, categoria e tipo.
  • Qualidade: uso de checklists antes de cada release.
  • Visibilidade: fácil acompanhamento do que está pendente, em andamento ou entregue.
  • Escalabilidade: aplicável a múltiplos sistemas derivados do MOLDE.

6. Recomendações Práticas

  • Comece com um Kanban simples (colunas: Pendente, Em Andamento, Bloqueado, Concluído).
  • Use Releases para agrupar entregas e gerar changelogs automáticos.
  • Crie Checklists padrão para tipos de entrega (publicação, segurança, performance).
  • Mantenha as enums (TipoItem, Prioridade, Status) no código para preservar regras de negócio.
  • Registre quem criou e quem executou cada Item de Evolução para rastreabilidade.

7. Como incluir este tutorial na documentação do MOLDE

  1. Adicione uma seção "Gestão de Desenvolvimento" no manual do MOLDE contendo este tutorial.
  2. Inclua exemplos concretos (cenários) e prints ou links para o Kanban e telas do app quando disponíveis.
  3. Mantenha o tutorial atualizado conforme o processo evolui (novas regras, novos checklists).

8. Próximos passos sugeridos

  • Implementar o Kanban no app (views + templates) para acompanhamento visual.
  • Adicionar filtros por Sistema, Categoria, Responsável e Versão Prevista.
  • Integrar (no futuro) com repositório Git para vincular commits/tags a Itens de Evolução.
  • Gerar relatórios: tempo médio de resolução, número de itens por categoria/versão, itens críticos abertos.
